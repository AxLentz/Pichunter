# 规则: Agent Rules 规范总纲 (Rules Guideline)

## 1. 命名格式标准

所有 Rule 文件必须遵循以下命名格式：

```
[序号]-[模式]-[业务名].md
```

### 示例
- `01-A-AgentBehavior.md`
- `02-A-ProjectSummary.md`
- `03-G-PythonStyle.md`

--------------------------------------------------

## 2. 模式与激活规则 (Activation Logic)

### 2.1 模式简写定义
| 简写 | 完整名称 | 含义 |
|---|---|---|
| **A** | Always On | 始终激活，作为操作底座 |
| **G** | Glob | 基于文件模式搜索激活 |
| **D** | Model Decision | 模型自主决定是否检索 |
| **M** | Manual | 仅在用户指引或 `@` 时加载 |

### 2.2 Glob 透明化原则
- 对于 **G 模式** 的规则，必须在文件内容的顶部（一级标题下方）显式注明其匹配路径（Glob Pattern）。

--------------------------------------------------

## 3. 命名与视觉规范

- **业务名**: 推荐使用 `Style`, `Summary`, `Behavior`, `Notes` 等实质词汇。
- **排版**: 使用 50 个短横线 `-` 作为模块分隔线。

--------------------------------------------------

## 4. 规则维护准则 (Maintenance Workflow)

- **元数据锁定**: 严禁 Agent 修改或覆盖 YAML 中的激活配置（UI 锁定）。
- **重大变动预告**: 在重排或调整整个 Rules 体系架构前，必须确认。
- **代码唯一真相 (SSOT)**: 涉及具体数据结构、接口字段或运行命令时，应引导 Agent 直接阅读代码定义（如 `package.json`、源码定义），避免在 Rule 文件中重复定义易变的信息。

--------------------------------------------------
